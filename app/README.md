# lapochka mini app

Telegram Mini App для бренда lapochka, реализованный pixel-perfect по макету
Figma [`lapochka-mini-app`](https://www.figma.com/design/NMlgJS4jP5vFIj2l9zfhmA/lapochka-mini-app?node-id=1-124)
(фрейм `Main`, node `1:124`).

Стек: **React + TypeScript + Vite + CSS Modules + React Router**, архитектура —
**Feature-Sliced Design**, интеграция с Telegram — через изолированный слой
`shared/telegram`. Без Tailwind и UI-библиотек — все компоненты написаны с нуля.

## Архитектура

Проект следует **Feature-Sliced Design**:

```
src/
├── app/            — точка сборки приложения
│   ├── providers/  — TelegramProvider, AppProviders
│   ├── router/     — AppRouter (React Router)
│   └── styles/     — global.css, reset.css (импортируют shared/styles/tokens.css)
├── pages/          — по одной странице на каждый экран
│   ├── HomePage/       — экран "Main" из Figma (1:1)
│   ├── CatalogPage/    — каталог вкусов (см. "Допущения")
│   ├── FlavourPage/    — карточка вкуса (см. "Допущения")
│   └── NotFoundPage/
├── widgets/        — крупные блоки компоновки страницы
│   ├── AppTopBar/, Hero/, MissionBanner/, FlavourCarousel/,
│   └── TreasureMap/, MomentsGallery/, BottomNavigation/
├── features/       — самостоятельные пользовательские сценарии
│   └── firstPurchasePromo/   — модалка "первая покупка" (ui + хук состояния)
├── entities/       — бизнес-сущности: ui + модель + мок-данные
│   ├── flavour/    — types.ts, mock.ts, api.ts, ui/FlavourCard
│   ├── moment/     — types.ts, mock.ts, api.ts, ui/MomentTile
│   └── mapFinding/ — types.ts, mock.ts, api.ts
└── shared/         — переиспользуемые кирпичи без бизнес-логики
    ├── ui/components/   — Button, IconButton, Skeleton, EmptyState, ErrorState
    ├── telegram/        — ЕДИНСТВЕННАЯ точка входа в Telegram WebApp SDK
    ├── icons/           — иконки (SVG)
    ├── config/          — routes.ts
    └── styles/          — tokens.css (дизайн-токены)
```

Правило зависимостей соблюдено во всём проекте:
`pages → widgets/features → entities → shared`. Слой не импортирует ничего
выше себя (это же проверяется глазами — все `import` идут по относительным
путям строго "вниз" по этой цепочке).

## Структура проекта

Каждый компонент — отдельная папка с обязательным `index.ts`:

```
ComponentName/
├── ComponentName.tsx
├── ComponentName.module.css
└── index.ts
```

Каждая сущность (`entities/<name>`) имеет:

```
entities/<name>/
├── ui/…                — презентационные React-компоненты
├── model/
│   ├── types.ts         — интерфейс сущности
│   ├── mock.ts           — мок-данные, повторяющие форму будущего API
│   └── api.ts             — функции fetchX(), уже сегодня асинхронные
│                             (Promise + имитация сетевой задержки),
│                             чтобы страницы уже сейчас показывали
│                             loading/empty/error как с настоящим бэкендом
└── index.ts              — публичный интерфейс слайса
```

## Запуск локально

Требуется Node.js 20+.

```bash
npm install
npm run dev
```

Приложение поднимется на `http://localhost:5173/mini-app/` (базовый путь
`/mini-app/` захардкожен в `vite.config.ts` под текущий GitHub Pages деплой —
поменяйте `base`, если разворачиваете в другом месте).

Другие команды:

```bash
npm run build     # tsc -b && vite build — типы + прод-сборка в dist/
npm run preview   # локальный просмотр прод-сборки
npm run lint      # oxlint
```

## Запуск без Telegram

Приложение полностью работает в обычном браузере:

- `shared/telegram/telegram.ts` при отсутствии `window.Telegram.WebApp`
  автоматически подставляет `shared/telegram/mockTelegram.ts` — заглушку с
  тем же интерфейсом (MainButton, BackButton, HapticFeedback, themeParams,
  initDataUnsafe.user).
- Ни один компонент, хук или страница не проверяют "мы в Telegram или нет" —
  они просто вызывают хуки из `shared/telegram`, а разруливание реальный/мок
  происходит один раз, в одном файле.
- В консоли браузера при локальном запуске появится
  `[shared/telegram] Telegram WebApp SDK not found — falling back to mockTelegram…`
  — это ожидаемо и не является ошибкой.

## Подключение Telegram-бота через BotFather

1. В Telegram напишите [@BotFather](https://t.me/BotFather) → `/newbot`
   (или используйте существующего бота, например `@lapochka_asha_bot`).
2. `/mybots` → выберите бота → **Bot Settings → Menu Button → Configure Menu Button**
   → укажите URL опубликованного приложения (см. следующий пункт) и текст
   кнопки, например «Открыть lapochka».
3. Либо через `/newapp`, если нужен полноценный Mini App с собственной
   иконкой и описанием в профиле бота.
4. Токен бота **не используется на фронтенде** — он нужен только бэкенду для
   валидации `initData` (см. `VITE_TELEGRAM_BOT_TOKEN` ниже и раздел
   "Где менять API").

## Настройка Mini App и ngrok

Для разработки с реальным Telegram-клиентом нужен HTTPS-адрес:

```bash
npm run dev
# в отдельном терминале:
npx ngrok http 5173
```

Полученный `https://xxxx.ngrok-free.app` укажите в Menu Button (см. выше)
вместо продакшен-адреса на время разработки. Не забудьте открыть путь с
базовым префиксом, если он задан в `vite.config.ts` (`base`).

Для продакшена: `npm run build`, задеплойте содержимое `dist/` (например, на
Cloudflare Pages/GitHub Pages) и укажите этот URL в BotFather.

## Где менять API после подключения backend

Всё мок-состояние живёт в `entities/<name>/model/`:

- `mock.ts` — статичные данные.
- `api.ts` — функции `fetchX()`/`fetchXById()` с той же сигнатурой
  (`Promise<T>`), что будет и у реального запроса.

Чтобы подключить бэкенд, меняется **только** тело функций в `api.ts`
(например, `fetch('/api/flavours').then(r => r.json())` вместо
`delay(flavours)`) — виджеты и страницы, которые их вызывают, не меняются
вообще, так как работают с тем же `Promise<Flavour[]>`.

Telegram `initData` уже доступен через `useTelegramUser()` — на бэкенде его
нужно валидировать (HMAC подписью бота) перед тем, как доверять
`user.id`; см. `shared/telegram/hooks/useTelegramUser.ts`.

## Дизайн-система

Токены вынесены в `shared/styles/tokens.css` и подключены глобально через
`app/styles/global.css`. Системные цвета Telegram (`--tg-theme-*`) всегда
имеют приоритет — `useTelegramTheme()` перезаписывает их на `document.documentElement`
при получении `themeChanged`; вне Telegram используются точные значения из
Figma-макета как фолбэк.

| Токен | Значение | Где взято |
|---|---|---|
| `--color-text-primary` | `#393838` (или `--tg-theme-text-color`) | текст заголовков |
| `--color-text-secondary` | `#696969` (или `--tg-theme-hint-color`) | вторичный текст |
| `--color-coral` | `#FE7962` (или `--tg-theme-button-color`) | акцент, CTA, пины |
| `--color-coral-light` | `#FFE8E2` | фон вторичной кнопки |
| `--color-surface-muted` | `#F5F4F4` | карточка "Миссия дня" |
| `--color-mission-badge` | `#9DB75B` | бейдж таймера миссии |
| `--fs-display / h1 / h2 / body / small` | `76 / 24 / 20 / 16 / 14px` | заголовок Hero → подписи |
| `--space-1…10` | `4–40px`, шаг 4px | вся вёрстка |
| `--radius-sm…2xl, full` | `12–32px, 999px` | карточки, кнопки, модалка |
| `--shadow-card / pin / can` | см. файл | нижняя навигация, пины на карте, банка |

Бренд-палитра из `product_book.md §12` (`--color-brand-yellow #F5C842`,
`--color-brand-orange #FF6B35`, `--color-brand-cream #FFFDF0`) сохранена в
токенах для будущих экранов, но не используется в текущем экране `Main`, где
Figma задаёт другую, более светлую палитру (см. "Допущения").

## Экраны и состояния

| Экран | Маршрут | Состояния |
|---|---|---|
| Главная (`HomePage`) | `/` | default; модалка "первая покупка" (auto-open / closed) |
| Каталог (`CatalogPage`) | `/catalog` | loading (skeleton-сетка), success, empty, error (с повтором) |
| Карточка вкуса (`FlavourPage`) | `/catalog/:flavourId` | loading (skeleton), success, not-found, error |
| 404 (`NotFoundPage`) | `*` | — |

Внутри `HomePage` каждый виджет с асинхронными мок-данными
(`FlavourCarousel`, `TreasureMap`, `MomentsGallery`) самостоятельно проходит
через `loading → success | empty | error`, с retry-кнопкой в `ErrorState`.

Кнопки и карточки имеют состояния `default`/`:active` (масштаб/фон),
`:disabled`, `:focus-visible` (видимый фокус-ринг для клавиатурной
навигации — см. `app/styles/reset.css`).

## Доступность

- Семантические теги: `<nav>`, `<main>`, `<header>`, `<h1>/<h2>`, `<time>`.
- У всех декоративных изображений `alt=""` + `aria-hidden`; у содержательных
  (банка вкуса, карта, фото) — осмысленный `alt`.
- Все интерактивные элементы — нативные `<button>`/`<a>` с `aria-label`, где
  нет видимого текста.
- Touch target ≥ 44×44px у всех кнопок (см. `IconButton`, `BottomNavigation`).
- Видимый `:focus-visible` для клавиатурной навигации по всему приложению.
- Учтён `prefers-reduced-motion` — анимация модалки/скелетонов отключается.

## Telegram-специфика

- `#root`/`body` используют `100dvh` — корректно во вьюпорте Telegram.
- `padding-bottom: max(16px, env(safe-area-inset-bottom))` у нижней навигации
  и модалки — не залезают под системные жесты/панели.
- `shared/telegram/telegram.ts::initMiniApp()` вызывает `ready()` + `expand()`
  один раз при старте (`TelegramProvider`).
- `useMainButton`/`useBackButton` управляют нативными кнопками Telegram
  (пример — `FlavourPage` показывает нативный MainButton «Хочу этот вкус!»).

## Допущения

Так как макет Figma (`lapochka-mini-app`, node `1:124`) содержит только один
фрейм — `Main` (главный экран), при выполнении требования "все экраны из
макета" и "добавь роутинг между экранами" были приняты следующие
продуктовые решения:

1. **Каталог вкусов (`/catalog`) и карточка вкуса (`/catalog/:id`)** — экраны
   построены дополнительно, так как шеврон рядом с «Что попробуем сегодня?»
   и сами карточки вкусов явно предполагают переход куда-то дальше. Стиль и
   токены взяты из того же дизайн-языка (та же типографика, радиусы, цвета),
   но точной раскладки в Figma для них нет — это осознанное расширение, а
   не «додуманный» пиксель-перфект.
2. **Нижняя навигация**: реально работают только «Главная» и «Каталог»
   (иконка списка) — единственные существующие экраны. Иконки «Карта»,
   «Добавить», «Профиль» визуально соответствуют макету, но ведут в никуда
   (дают лёгкую haptic-обратную связь) — заглушки под будущие экраны.
3. **Модалка «Твои первые лапки уже ждут»**: в макете нет явного триггера
   открытия. Решено показывать её автоматически один раз на устройство
   (флаг в `localStorage`) через ~1.2с после захода на главную, с
   возможностью закрыть и больше не показывать.
4. **Таймер «Миссия дня»**: в макете это статичная надпись `20:35:02`.
   Реализован как настоящий обратный отсчёт (тикает раз в секунду и
   зацикливается по достижении нуля) — бизнес-правило сброса миссии не
   описано, поэтому выбран самый очевидный вариант "обратный отсчёт до
   следующей миссии".
5. **Кнопки-«капли» (`Button`)**: в Figma это органическая (не
   прямоугольная) форма с двумя цветами заливки (`#FE7962` и `#FFE8E2`).
   Вместо CSS `border-radius` (даёт другую форму) переиспользуется
   исходный SVG-контур как фон кнопки — это позволяет остаться в рамках
   "без UI-библиотек, компонент написан самостоятельно" и сохранить точную
   форму.
6. **Раздел "карта находок" в фоне hero (`Group 3039`, декоративная сетка
   линий)** визуально почти не заметен на самом макете (эффект
   `Subtract` поверх полупрозрачной сетки) и не влияет на пиксельное
   соответствие — не воспроизведён, чтобы не добавлять шум поверх фото.
7. **Дублирующаяся карточка «Персик + Вербена»**: в макете `Frame 2087327384`
   действительно два инстанса `CardLemonade` с одинаковым названием, но
   разными фото — перенесено как есть (два разных `id` в моках, одно и то
   же отображаемое имя).
8. **`@telegram-apps/sdk`**: вместо тяжёлого сигнального SDK-пакета
   выбрана тонкая самописная обёртка (`shared/telegram`) поверх
   официального `telegram-web-app.js` (подключается в `index.html`,
   как рекомендует Telegram) — она проще, полностью прозрачна для аудита
   ("нет прямых вызовов SDK вне `shared/telegram`" проверяется одним
   grep'ом) и не требует дополнительной инициализации сигналов/промисов
   монтирования компонентов, что укладывается в рамки MVP на 1 день. При
   необходимости `shared/telegram/telegram.ts` — единственное место,
   которое нужно переписать на `@telegram-apps/sdk`, без изменений выше
   по слоям.
9. **Бренд-палитра `product_book.md §12`** (`#F5C842`/`#FF6B35`/`#FFFDF0`)
   не совпадает с палитрой конкретно этого экрана в Figma (белый фон,
   `#FE7962` коралл). Обе палитры сохранены как CSS-токены; фактический
   экран `Main` рендерится по Figma-точным цветам, бренд-токены оставлены
   про запас для будущих экранов, где могут понадобиться жёлтый/оранжевый.
